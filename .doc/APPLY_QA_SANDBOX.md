## /apply 에이전트 온보딩 QA & 샌드박스 가이드

### 1. QA 전제 조건

- Admin 계정이 있고, 최소 1개 지점에 대해:
  - `/admin/onboarding` 또는 `/admin/branch` 에서 **이름/지점 설정** 완료.
  - `/admin/invite-codes` 에서 **초대코드 1개 이상 생성**.
- 브라우저에서 GA NEXUS dev 서버가 떠 있는 상태:
  - 예: `npm run dev` 후 `http://localhost:3000`.

---

### 2. 기본 플로우 샌드박스 시나리오

#### 시나리오 A – 정상 온보딩

1. Admin 로그인 → `/admin/invite-codes` 에서 새 초대코드 발급.  
   - 예: `MERITZ_GANGNAM-ABC123`.
2. 시크릿/다른 브라우저 창 열기 → `http://localhost:3000/apply` 접속.
3. **Step1 – 초대코드 입력**
   - 초대코드 필드에 `MERITZ_GANGNAM-ABC123` 입력.
   - “코드 확인” 클릭 → 정상일 경우 Step2로 진입, 상단 안내에 지점명 노출되는지 확인.
4. **Step2 – 기본 정보 입력**
   - 성함: 임의 값 (예: `테스트 에이전트`).
   - 소속 지점: 읽기 전용, 초대코드 지점명과 일치하는지 확인.
   - 생년월일: `950101` 등 6자리.
   - 휴대폰 번호: `01012345678` 입력 → 자동 포맷 `010-1234-5678` 확인.
   - 매니저 코드: 임의 문자열 (백엔드에서는 선택 필드로 수신).
   - “[필수] 개인정보 수집…” 체크.
   - “신청서 제출” 클릭 → `POST /api/agent/apply` 성공 시 Step3로 이동하는지 확인.
5. **Step3 – 승인 대기 화면**
   - “신청이 완료되었습니다. 지점장님의 승인 후 메인 캘린더에 접근할 수 있습니다.” 문구 확인.
   - GA_NEXUS DB에서:
     - `profiles`에 `role='agent'`, `is_approved=false`, `branch_name=초대코드 지점` 으로 레코드 생성됐는지 확인.
     - `auth_users`에 새 에이전트 계정이 생겼는지 확인(초기 PW=생년월일).

#### 시나리오 B – 잘못된 초대코드

1. `/apply` 접속.
2. Step1에서 존재하지 않는 코드 입력 (예: `INVALID-CODE`).
3. “코드 확인” 클릭 시:
   - 붉은 에러 배너로 서버 메시지(예: “유효하지 않은 초대 코드입니다.”)가 노출되는지 확인.
   - Step2로 넘어가지 않고 그대로 Step1 유지되는지 확인.

#### 시나리오 C – 필수 정보 누락

1. 유효한 초대코드로 Step2까지 진입.
2. 성함, 생년월일, 휴대폰, 동의 체크 중 하나 이상 비운 상태에서:
   - “신청서 제출” 버튼이 비활성/오류 메시지로 막히는지 확인.

---

### 3. 에러/엣지 케이스 샌드박스

#### C-1. 동일 초대코드로 여러 번 신청

1. 동일 초대코드를 사용해 `/apply` 온보딩을 2~3회 반복.
2. `invite_codes.used_count` 증가와 API 레벨에서 설정한 `max_uses` 정책 동작(있다면)을 확인.
3. `max_uses` 초과 시:
   - `/api/agent/apply`에서 적절한 에러 메시지(예: “이 초대 코드는 더 이상 사용할 수 없습니다.”)가 응답되는지,
   - `/apply` Step2에서 에러 배너로 표시되는지 확인.

#### C-2. 네트워크 오류 의도적 유발

1. dev 서버를 잠시 내려(`Ctrl+C`) 둔 상태에서 Step1 또는 Step2에서 버튼 클릭.
2. “네트워크 오류로 … 할 수 없습니다.” 형태의 사용자 친화적인 에러 메시지가 나오는지 확인.

---

### 4. QA 체크리스트 (요약)

- [ ] 유효한 초대코드로 Step1 → Step2 전환이 잘 된다.
- [ ] 잘못된 초대코드 입력 시 Step1에서 에러만 표시되고 넘어가지 않는다.
- [ ] Step2에서 필수 필드(성함/생년월일/휴대폰/동의)가 비어 있으면 제출이 막힌다.
- [ ] 정상 제출 시 DB의 `auth_users` / `profiles` / `invite_codes`(used_count)가 기대대로 갱신된다.
- [ ] 성공 후 Step3에서 승인 대기 안내가 노출된다.
- [ ] 동일 코드 재사용 또는 초과 사용 시, 서버/클라이언트 모두 에러 처리 흐름이 자연스럽다.
